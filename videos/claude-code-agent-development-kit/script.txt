Claude Code has 5 customization layers. Most engineers only touch one. Here is what the other four unlock. And how to know which one you actually need. By the end of this video, you'll have a one-screen mental model of all 5 layers. A decision tree that picks the right layer for any extension you want to build. And a map of how all 5 fire during a single request. This is not a getting-started tutorial. We're not installing anything. We're not showing prompts. We're going one level above. The architecture. Anthropic's docs call this an extension model. The community started calling it something better. The Agent Development Kit.

5 layers. 60 seconds. Layer 1 is CLAUDE.md. Always-on facts that load into every request. Your project constitution. Layer 2 is Skills. On-demand knowledge. Claude reads a one-line description, and pulls the full body only when relevant. Layer 3 is Hooks. Deterministic guards. Shell commands or HTTP calls that fire at lifecycle events. 31 events. 5 handler types. Layer 4 is Subagents. Forked workers with their own context window, their own model, their own tools. They return only a summary. Layer 5 is Plugins. The distribution wrapper. Bundle skills, agents, hooks, and MCP into one install. Whole team gets the same behavior. And MCP sits next to all 5. Not a layer. A sidebar. The wire to external systems. Each of these solves a different problem. Confuse them and you'll build the wrong thing.

Layer 1. CLAUDE.md. It's a markdown file injected as a user message at session start. Project-level. User-level. Organization-level. They all stack together. Every line loads into every request. That's the price of being always-on. So Anthropic recommends keeping each file under 200 lines. Past that, adherence degrades. Claude starts skimming. The mistake most teams make is treating CLAUDE.md as a wiki. Multi-step procedures. Reference documentation. Edge-case notes. All of it crammed in. That content still loads on requests that don't need it. So you pay the token cost on every turn for documentation you only use once a week. The right model is a constitution. Project layout. Build commands. Naming conventions. Never-do-X rules. Always-do-Y rules. Things that apply to every session. If Claude makes the same mistake twice, that goes in CLAUDE.md. If you find yourself pasting the same prompt over and over, that's not a CLAUDE.md line. That's a skill. New in Claude Code version 2. The instructions-loaded hook tells you which CLAUDE.md files actually loaded for a session. And auto-memory now writes its own MEMORY dot md file under tilde slash dot claude, so context survives across conversations.

Layer 2 and Layer 3. They look similar. They are not. Skills are guidance. Hooks are enforcement. A skill is a directory with a SKILL.md file. YAML frontmatter on top. Markdown body underneath. Only the description loads at session start. The body loads on demand when Claude matches the description to your request. Or when you invoke it explicitly with slash and the skill name. Description budget is 1536 characters per skill. About 1 percent of your context window in total. That's how you ship 100 skills without bankrupting context. The full body only enters context when Claude actually needs it. A hook is different. It runs deterministically at a lifecycle event. There are 31 of those events in Claude Code version 2. Session start. User prompt submit. Pre-tool use. Post-tool use. Stop. And 26 others. 5 handler types. Shell command. HTTP call. MCP tool. Prompt. Subagent. The key difference. Writing "never edit dot env" in a skill is a request. Claude can reason around it. A pre-tool-use hook that exits with code 2 when the path matches dot env. That's a guarantee. It cannot be reasoned around. Skills can also fork into their own context window when they're heavy. Set context fork in the frontmatter. Now your skill runs in an isolated subagent and never pollutes the main thread. Skills and hooks compose. A post-tool-use hook runs lint after every edit. If lint fails, the error feeds back into Claude. A slash fix-lint skill then tells Claude how to remediate. Enforcement plus guidance, working together.

Layer 4. Subagents. This is where context isolation enters the kit. A subagent is a markdown file at dot claude slash agents slash some name. Frontmatter says what model, what tools, what permissions. The body is a custom system prompt. When the main conversation asks for something matching the subagent's description, Claude forks. The subagent gets its own context window. Its own model. Its own tool list. Its own permission mode. It does the work. Then it returns only a summary back to the main thread. This solves two problems at once. First, context pollution. If you need to read 50 files to answer one question, you don't want all 50 files in your main conversation. The subagent reads them in its own window. It returns 3 sentences. Second, cost routing. The main thread can be Opus. The subagent that reads those 50 files can be Haiku. Same answer. Different bill. Three subagents ship built in. Explore. Haiku, read-only, fast codebase search. Plan. Inherits the model, read-only, used in plan mode. And general-purpose. Inherits everything. You can also run subagents in isolation worktree mode. The subagent gets a temporary git worktree. A real copy of the repo, isolated from your working tree. It can edit, test, commit. Nothing touches your branch until you choose to merge. No recursion, by the way. Subagents cannot spawn other subagents. That cap is intentional.

Layer 5. Plugins. The wrapper that turns Layer 1 through 4 into a deliverable. A plugin is a directory with a plugin dot json manifest. Inside, you can put any combination of skills. Subagents. Hooks. MCP servers. LSP servers. Custom commands. One install command. Whole team gets the same configuration. Versioned. Namespaced. Skills inside a plugin become slash plugin-name colon skill-name automatically. So short names don't collide across plugins. Marketplaces are how plugins travel. The official Anthropic marketplace lives at the anthropics slash claude-code GitHub repo. Private team marketplaces use private git. One repo. One install. Same agent capabilities across the whole organization. This is how a personal workflow becomes a team standard. One thing to know. Plugin subagents drop hooks, MCP servers, and permission mode for security. If a subagent needs those, copy it into dot claude slash agents instead of shipping it in a plugin.

Decision tree. 5 questions. Pick the first yes. Question 1. Should this load on every request? Yes. CLAUDE.md. Question 2. Does this connect to an external system. Postgres. Slack. GitHub. Figma. Yes. MCP. Note that's the sidebar. Not a layer. Question 3. Must this run deterministically at a specific lifecycle event? Yes. Hook. Question 4. Does this need its own isolated context. Its own model. Its own permission scope? Yes. Subagent. Question 5. None of the above? Default. Skill. And if you want all of that to travel with your team, wrap whatever you built into a plugin. That's the sixth question. Distribution. Five questions get you to the right layer. The sixth question moves it to the team.

Here's how the kit fires on a single request. CLAUDE.md is there from the first token. Always. The user prompt comes in. The user-prompt-submit hook fires. Claude scans skill descriptions. If one matches, the skill body loads. Claude reaches for a tool. The pre-tool-use hook fires. Tool runs. The post-tool-use hook fires. If the task needs isolated context, Claude forks a subagent. The subagent does its work in its own window. Returns a summary. Main thread continues. Plugins wrap whichever of these pieces you bundled, so they all show up at install time. That's the kit. 5 layers. One sidebar. One distribution mechanism. Most engineers really do touch only Layer 1. The other four are where production agents are built.
