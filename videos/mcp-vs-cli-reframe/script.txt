For most of twenty twenty-five, AI engineers were yelling at each other. One side: just give the agent a shell. The other: use M C P, the protocol Anthropic shipped last November. Both sides ran benchmarks. Both sides had screenshots. Both were arguing about the wrong thing. We probed seven real M C P servers ourselves. Sixteen thousand tokens of schema sitting in the model's prompt before the agent does anything useful. In the next eight minutes you'll see the one move that took the same workflow from one hundred and fifty thousand tokens down to two thousand. Where the debate actually came from. And what to do about it on your own machine. Let's go.

This is where it started. In November twenty twenty-four, Anthropic published the Model Context Protocol — a standard way for agents to talk to external tools. Within twelve months, two camps had formed. The M C P camp pushed typed contracts, O auth, audit trails, multi tenant safety. The C L I camp said the protocol was paying tax on knowledge the model already had. The fights looked like they were about M C P versus C L I. They weren't. They were about how tool definitions get loaded into the model's context window. That distinction is the whole story.

Here's what each side actually measured. The C L I camp ran the numbers. Playwright's M C P server eats around fifteen thousand tokens of schema before it does anything. Chrome DevTools eats about seventeen thousand. A typical multi server setup burns fifty thousand tokens of context before the agent runs its first command. We reproduced this on this machine. Seven real M C P servers. One hundred and one tools. Sixteen thousand five hundred tokens of pure schema, sitting in the model's prompt at session start. Now the M C P camp's side. C L I agents break on multi tenant apps because there's no shared state. There are no typed contracts, so the agent guesses at output shapes. On unfamiliar APIs, agents waste turns parsing raw H T M L. IBM showed an agent reverse engineering Next dot J S internals just to read a webpage. We ran that experiment too. Plain curl on a single page app returns one thousand seven hundred tokens of empty H T M L — no rendered content. A Playwright M C P server, with a real headless browser, returns one thousand eight hundred and eighty nine tokens — the full page. Both sides had receipts. Both arguments were real. So which one wins? Neither. That was the wrong question.

Here's the move that resolved it. On November fourth, twenty twenty-five, Anthropic published an article called Code Execution with M C P. Their example was simple. Pull a Google Drive transcript. Push a summary to Salesforce. The old way, with both tool schemas eagerly loaded and the transcript piping through the model on every step, burned one hundred and fifty thousand tokens. The new way, a few lines of TypeScript that import only what they need, burned two thousand. A ninety eight point seven percent drop. Three months later, Cloudflare took it further. They exposed their entire two thousand five hundred endpoint A P I through M C P — using just two functions. Search, and execute. The traditional version would have consumed one point one seven million tokens of schema. The new version: about one thousand. Ninety nine point nine percent reduction. The fix wasn't a different protocol. The fix was a different binding. Tool definitions stopped living in the model's context. They moved to the filesystem.

So what is Code Mode, in plain language? The model writes code. The code calls tools. The tools live on disk. Five layers, top down. The agent on top, picks the right primitive per task. Below it, the runtime executes that code in a sandbox. Underneath the runtime, bash gives you everything already on your path — git, curl, grep, ffmpeg. The model has seen these in training. No schema needed. Beside bash, typed module imports give you the proprietary APIs — Stripe, Salesforce, your internal services. Each tool is a small TypeScript file. The type signature travels with the import. Tool definitions enter context only on the import line. Every other tool stays out. And underneath all of it, M C P still sits there — for the cases where you actually need multi tenant O auth, runtime tool discovery, or audit trails. Picture it. Three lines. Import search files from at tools slash github. Import send message from at tools slash slack. Two function calls. The Github and Slack tool definitions enter the prompt only on the import lines. Every other tool in the catalog stays out. M C P's typed contracts, plus C L I's lazy loading, in one runtime.

You'll see takes online that M C P is dying. They're wrong. Anthropic just reported ninety seven million monthly S D K downloads. Over ten thousand active public M C P servers. Seventy five plus Claude connectors built on it. This is the fastest growing piece of agent infrastructure right now. The verified sources are right there. Anthropic's Code Execution article from November fourth, twenty twenty-five. Cloudflare's Code Mode post from February twentieth, twenty twenty-six. Anthropic's own Agent Skills documentation, which ships the same idea inside Claude — three level progressive disclosure. Just the frontmatter loads by default, about thirty to fifty tokens per skill. The full instructions only load when the skill is relevant. Linked files only when the skill body references them. That's Code Mode for the agent on your laptop, today, no extra infrastructure. What died was the assumption that you load every tool upfront. It's also true that some companies are stepping back. In March twenty twenty-six, Perplexity moved away from M C P for their flagship Agent A P I, citing context bloat and authentication friction. They kept their M C P server for third party developers though. They dropped it for their own product because their use case is single agent, known tools, predictable workflows. That's the opposite of where M C P shines.

So what do you do about all this in your own setup? Three rules. If a binary on your path already does the job, use bash. It's free. The model knows it. If you're calling a proprietary A P I, write a typed module import the agent can pull in on demand. If you genuinely need runtime tool discovery, multi tenant auth, or governance, use M C P — but expose it through a search and execute pattern like Cloudflare did. Don't load the full schema at session start. Two things nobody on either side has solved cleanly yet. Cross server composition. Cloudflare's pattern works beautifully for one A P I. If your agent needs to compose five different services, you're back in context pressure territory unless every one ships as a code import. And discoverability. With M C P, the agent gets the menu of available tools for free at session start. With Code Mode, the agent has to know which imports exist. Anthropic Skills solves it through filesystem layout. Cursor and Cline solve it differently. There's no standard yet. And the tradeoff isn't free. Code Mode moves the cost from the context window to the sandbox. For a developer running Claude Code locally, that cost is negligible — it's your machine. For an enterprise running thousands of agents, it's a real engineering line item. Here's the takeaway. M C P versus C L I was the wrong frame. M C P isn't dying — it's getting demoted from the runtime to a primitive the runtime composes. Bash for binaries. Typed imports for proprietary APIs. M C P underneath when you need contracts and auth. The agent picks per task. That's the answer.
