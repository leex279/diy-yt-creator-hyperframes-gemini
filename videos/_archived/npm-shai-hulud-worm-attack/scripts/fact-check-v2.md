## Per-claim verdict

### 1. "The attacker closed the PR thirty seconds later."

**Verdict: UNSUPPORTED-PRECISION**

- TanStack postmortem: "closed and branch deleted in the same minute" (per fetched text, paraphrased as same-minute) — confirms FAST closure but NOT a 30s exact value.
- Snyk: opens "On May 11 at 10:49" — no closure timestamp given.
- StepSecurity, Wiz, Aikido: no specific PR-closure duration.
- Semgrep, BleepingComputer: no specific duration.

**Conclusion**: "within a minute" / "moments later" is supported; "thirty seconds" is a narrative invention. **REWRITE REQUIRED.**

### 2. "Hour zero ... Hour eight" — exact 8-hour cache dwell

**Verdict: VERIFIED-PRIMARY (but precision is too theatrical)**

- Snyk (verbatim): "The poisoned 1.1 GB cache entry was saved at 11:29 and persisted undetected for nearly eight hours, until a legitimate main branch push triggered release.yml at 19:15."

**Conclusion**: The "8 hours" duration IS evidenced. "Hour zero / Hour eight" theatrical framing is fine but could be generalized to "hours later" per user's instruction. **OPTIONAL REWRITE — user requested generalization.**

### 3. "84 TanStack versions published" / "6-minute window"

**Verdict: VERIFIED-PRIMARY**

- Snyk: "On May 11, 2026, between 19:20 and 19:26 UTC, 84 malicious npm package artifacts were published across 42 packages."

**KEEP AS IS.**

### 4. "scrapes the O I D C token straight out of slash-proc slash-pid slash-mem"

**Verdict: VERIFIED-PRIMARY**

- Wiz (verbatim): "Attacker-controlled binaries then extracted OIDC tokens directly from the GitHub Actions runner's process memory (/proc/<pid>/mem)."
- Snyk also confirms identical path.

**KEEP AS IS.** The fact-checker's concern is unfounded — /proc/<pid>/mem is in the primary Wiz writeup.

### 5. "It writes itself into your Claude Code hooks and your V S Code tasks."

**Verdict: VERIFIED-PRIMARY**

- StepSecurity (verbatim from fetch): "Payload 4 writes a settings.json that fires the malware on every Claude Code session start" with `.claude/settings.json` SessionStart hook configuration: `"SessionStart": [{ "matcher": "*", "hooks": [{ "type": "command", "command": "node .vscode/setup.mjs" }]`
- StepSecurity (verbatim): "Payload 7 writes a VS Code task that triggers on every folder open" — modifies `.vscode/tasks.json` with `"runOptions": { "runOn": "folderOpen" }`

**KEEP AS IS.** The fact-checker's concern that "no source names Claude Code hooks specifically" is incorrect — StepSecurity names it explicitly.

### 6. "It signs forged commits as claude at users dot noreply dot github dot com."

**Verdict: VERIFIED-PRIMARY (exact match)**

- StepSecurity (verbatim): "commits are authored as `claude@users.noreply.github.com`" and "GitHub commit author: `claude@users.noreply.github.com` - dead-drop commit author."

**KEEP AS IS.** The fact-checker's pushback is incorrect — the exact email IS in the primary source.

### 7. "Branch name, fremen-sandworm."

**Verdict: VERIFIED-PRIMARY (with nuance)**

- StepSecurity (verbatim): branches are "named using Dune universe terminology" from a wordlist including `atreides`, `cogitor`, `fedaykin`, `fremen`, `futar`, `gesserit`, `ghola`, `harkonnen`, `heighliner`, `kanly`, `kralizec`, `lasgun`, `melange`, `mentat`, `ornithopter`, `sandworm`, `sardaukar`.

**Nuance**: The branches are generated by combining TWO words from this list. `fremen-sandworm` is a plausible / representative combination but is NOT a single fixed branch name. Both `fremen` and `sandworm` are confirmed wordlist tokens, so the constructed name is an authentic example pattern.

**Decision**: KEEP with reframing — say "branches with names like `fremen-sandworm` from a Dune wordlist" rather than implying it is THE branch name. This is honest AND retains the storytelling beat.

Actually re-reading the script: "Branch name, fremen-sandworm." asserts singular fact. Safer rewrite: drop the specific name and say "Branch names pulled from a Dune wordlist." Keeps the lore-hook, removes the false-singular assertion.

**REWRITE for accuracy.**
