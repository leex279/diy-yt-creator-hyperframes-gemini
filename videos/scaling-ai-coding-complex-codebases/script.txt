Last Friday, the Dynamous community ran a ninety-minute interactive workshop with Cole Medin on something most AI coding content gets wrong. The codebase. Ninety-seven Dynamous members in the room. Six strategies. This video is the quick summary.

Here's the problem. Vibe coding doesn't scale. Every team starts simple. A clean greenfield. One Claude Code session. Smooth shipping. But it always ends in two-thousand-line god components. Agents hallucinating patterns you never showed them. Context bloat killing every session.

And one specific failure mode is killing people. A bad prime command. Some teams run a prime, and it pulls in tens of thousands of tokens before they've written a single line of code. The agent's context is its most precious resource. Pre-bloat it, and reasoning quality collapses. You're already losing.

Here's the pivot. File size isn't the right metric. A two-thousand-line file with three imports is fine. A five-hundred-line file imported by twenty others is a fire. Coupling, not lines.

Six strategies. Each one stands alone. Combined, they turn any messy codebase into something an agent can actually navigate. One — vertical slice architecture. Organize by feature, not by layer. Two — brownfield day one. Use parallel sub-agents to derive the AI layer the first time. Three — context management. Three tiers. Four — focused priming. One prime command per slice. Five — parallel worktrees. Multiple agents, isolated copies, zero collisions. Six — coupling over line count.

Strategy one. Vertical slice architecture. Most codebases are organized by layer. Controllers in one folder. Services in another. Repositories in a third. Tests way over there. That's fine for humans who already have the system in their head. It's terrible for a coding agent. To touch one feature, the agent chases that feature across four folders, burning tokens at every hop. Vertical slice flips it. One folder per feature, owning everything that feature needs. Models. Schemas. Routes. Service. Errors. Tests. All in one place. The agent loads the slice, has the full feature in context, ships. And here's the part most senior engineers get wrong. You don't refactor everything. Start with the next new feature. Build it in the vertical slice shape. Let the brownfield decay around it. Six months later, every new feature ships vertical slice, and you've quietly migrated.

Strategy two. Most engineers conflate two different scenarios into one word — brownfield. Type A — there's no AI layer yet. The agent walks in blind. You do this once, ever, per codebase. Spin up parallel explorer sub-agents. Five for small. Eight or ten for a sprawling monorepo. Each one investigates one slice. They converge into a single codebase-analysis markdown. Then a second round of builder sub-agents derive your global rules plus a customized command suite. Ten minutes. Done forever. Type B — the AI layer exists. New epic incoming. This is where you live ninety-five percent of the time. Every sprint reads the layer. Edits where reality has drifted. Ships. Loops back. And a question came up in the chat. How do you know if the AI layer is trustworthy? Run the fill-gaps audit. Same parallel sub-agents pattern. Different goal. The agent compares what the layer claims against the actual code. Surfaces drift. Closes gaps.

Strategy three. Context management. Every codebase has three structurally different kinds of context. Global. On-demand. And Layer Two. Global is auto-loaded every session. Tech stack. Code style. Must-know conventions. Keep this under three hundred lines. On-demand is loaded only when relevant. Pattern guides. Per-feature READMEs. Layer Two is the plan for this specific ticket. Loaded once. Discarded after. If you put everything in global, you pay a token tax every session for context that's irrelevant ninety percent of the time. And another chat question — markdown or HTML? Markdown wins. HTML burns tokens for layout you don't need.

Strategy four. Focused priming. A vanilla prime command tries to read the whole project. On a small codebase, manageable. On a five-hundred-thousand-line monorepo, you blow the context budget before the task starts. The fix is a family of prime commands. Each loads one slice. Prime-server loads the backend. Prime-client loads the frontend. Prime-components for shared UI. Prime-endpoint for adding a new A P I route end-to-end. Each command loads five to ten files. The agent comes online with the right context for one type of task. And who writes these prime commands? You let the brownfield-day-one skill generate them. From your codebase. Customized. Day one.

And the workflow inside every ticket is the same three-move loop. You plan. Vertical-slice the work. Write the spec in its own session. Sub-agents scout. Then Claude implements. Fresh session. Only the plan loaded. Then you validate. Lint. Type-check. Unit. End-to-end. Manual review. Five steps. Not eight. Not twelve. Five is the ceiling. Anything past that is over-engineering.

Strategy five. Parallel worktrees. Same wallet, faster delivery. Git worktrees give you N working directories from the same repo. Each on its own branch. Each isolated. Combine with multiple Claude Code sessions, and you have N parallel agentic streams. Two terminals. Two tickets. Two pull requests. No cross-talk. People assume parallel sessions equal N times the cost. Not really. The total work is the same total tokens. You're compressing wallclock by running concurrently. The constraint isn't your spend. It's how fast you can review and merge. Most teams find their bottleneck is the human reviewer, not the agent.

Strategy six. Component sizing by coupling, not by line count. The god-component anti-pattern is what kills your AI workflow. Three questions. One. Can you work on this file in isolation? Or do you always need to also load three other files to make sense of it? Two. Does the agent have clear boundaries? Or does every change ripple outside the file? Three. Is the public A P I obvious? If any answer is no, the component is too coupled. Regardless of how many lines. And finding unused code? AST-grep. It walks the codebase as an abstract syntax tree. Anything with no inbound references is dead. Pair it with your validate step.

And here's the principle that landed hardest. Anthropic's own guidance on Claude Code in large codebases points to the same pattern. Split exploration from editing. Your primary session edits files. Sub-agents take the context hit. The most precious resource when you're working with a coding agent isn't your wallet. It's the context. Protect it like a finite tank.
