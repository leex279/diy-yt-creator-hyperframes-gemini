And then there are the workflows people have actually settled on. Six patterns showed up repeatedly across the replies.

Pattern one. Markdown for agents only. H T M L for agents and humans.

The exact framing from a builder running a content system. Reviewing ideas, drafts, and a whole workspace end-to-end in markdown is — terrible U X. So markdown stays as the agent layer. H T M L becomes the human review layer.

Pattern two. Data visualization in H T M L. Everything else in markdown.

There is no comparison for data viz. But for hierarchical text, a clean markdown viewer is just as legible. And saves tokens.

Pattern three. Markdown for input. H T M L for output to the customer.

As everything — it depends. H T M L is great for output to the customer. M D is better for input. You must understand where you will use it.

Pattern four. M D X. The Markdown-plus-React format.

One commenter. H T M L is triple the size and harder to read and write. The solution is M D X. Markdown is small, fast, readable. M D X combines dynamic content and web components when needed.

Pattern five. Mermaid stays in markdown.

You would think H T M L would win here. But a markdown viewer with Mermaid support is way better. I don't want an L L M coding a custom network diagram when I already have a clean, standard, token-efficient way to view them.

Pattern six. The conspiracy take.

The framing — someone earning money on token spending suggests we spend more tokens.

This one showed up four separate times in the comments. It is worth taking seriously, even if you do not agree.
