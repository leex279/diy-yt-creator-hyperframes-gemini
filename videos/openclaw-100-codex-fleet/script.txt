One developer. A hundred AI agents reviewing every line of code he ships. Zero junior PR reviewers.

That's the setup Peter Steinberger casually posted to eight hundred forty-seven thousand viewers this week. One person. A hundred Codex agents running in the cloud. Every pull request, every issue, watched in parallel.

People started freaking out about the bill. He posted something different. And the something different is what nobody is talking about.

Here's the question he's actually trying to answer.

How would we build software in the future if tokens don't matter?

That's the line. Not, look how much we spend. Not, look how many agents we run. The whole project, the whole tool stack, the whole hundred-agent fleet, is a working bet on one assumption. The reason most teams don't build like this is that tokens still feel scarce. Steipete is acting like that scarcity is already gone.

And to do that, his team runs roughly ten automations at once. Codex reviews every PR and every issue. A bot called claw-sweeper hunts down old issues that quietly got fixed on main. Codex reviews every commit for security holes. Another instance de-duplicates issues and clusters them. Agents spin up cloud sandboxes to record before-and-after demos. Codex watches new issues and drafts the PR itself. Codex polices spam comments. Codex tracks performance regressions into Discord. Agents listen on team meetings and start work as features are discussed. And claw-patch reviews everything by semantic unit.

Okay, back to the fleet, and the first custom tool. It's called claw-sweeper. Open source, MIT licensed, lives under the OpenClaw GitHub org. The job is simple to describe. It reads every open issue in your repo, checks whether the fix already landed on main, and if it did, it closes the issue with the exact merge reference attached.

It runs in four lanes. Review proposes closures. Apply executes them with safety checks. Repair runs autofix and automerge through maintainer commands. Commit Review analyzes everything that hits main.

Here's the part that makes it credible. In one reference week, claw-sweeper scanned three thousand four hundred seventy-eight issues. It closed about four of them. That's a zero point one percent close rate. Which sounds tiny, until you remember it never sleeps. It never closes a maintainer's own issue, never touches protected labels, and re-fetches the GitHub state right before any change. Conservative by design.

But sweeping old issues is the easy part. The next tool is wild.

It's called crab-box. Install with brew install openclaw slash tap slash crab-box. Crab-box is a shared agent workspace control plane. The tagline is, a short-lived box for every run. Lease a machine, sync your code, run the task, release the machine. No long-lived dev boxes sitting around burning credentials.

The architecture is the interesting part. A Cloudflare Worker brokers all the secrets. The C L I on your laptop only carries a bearer token. The remote machines never hold cloud credentials at all. Backends include Hetzner, AWS, and Azure on the brokered side, plus E2B, Daytona, Blacksmith, and Semaphore on the delegated side. You can lease Linux, macOS on EC2 Mac, or Windows on AWS or Azure. All three operating systems stream over VNC.

And here is the use case that sent the post viral. An agent leases a fresh box. The agent logs into Telegram inside that box. The agent reproduces a bug, records the screen, captures the before and after of the fix, and posts the video as a comment on the pull request. The whole demo is generated by the same fleet that wrote the code.

So the sandbox runs the demo. The third tool reviews the code.

It's called claw-patch dot A I. Same pattern. Open source, MIT, made by the OpenClaw team. The trick is how it slices a repo before review.

Most AI code review tools see one giant codebase and choke. Claw-patch maps the repo into semantic units first. Routes for Next dot J S. Commands for N P M or Go or Rust or Swift. Packages. C L I scripts. Tests. Each unit gets its own entrypoints, owned files, context files, and associated tests. Then the review runs per unit, not per file.

The findings come back categorized. Bug, security, performance, docs gap, test gap, maintainability. Each one tagged with a severity and a confidence score and pointed at the exact lines of evidence. You get a markdown report and a JSON report. Clean handoff to the next agent.

Back in, and here's how all of it stacks together. Claw-sweeper at the top, closing the dead issues. Crab-box in the middle, leasing the machines that run the work. Claw-patch underneath, slicing the code into reviewable units. Codex sitting across all three, doing the actual reading and writing. Vercel's deep-sec, released two weeks before this post, handling the security side, with Codex configured for security covering whatever deep-sec doesn't.

That's it. That's the hundred-agent fleet. Three thin, well-scoped tools, plus Codex, plus deep-sec. Composable, not magical.

Which is why the reaction split into roughly five camps.

Camp one is the dry skeptics. Steipete called the operation extremely lean. A hundred parallel agents and three custom-built tools is, by any normal definition of the word, the opposite of lean. The pushback is wry, not hostile. The dictionary needs an update.

Camp two is the forward-looking. Yes, only a handful of orgs can run this in 2026. But the cost of intelligence keeps falling. In eighteen to thirty-six months, this kind of fleet is the default for most teams. Steipete agreed with this take publicly.

Camp three is the cost optimizers. The pragmatic engineer reply. Have you tried Cerebras? Grok? Qwen? Deterministic hooks instead of LLM hooks? You could trim the bill dramatically. Some of these suggestions are genuinely good. Steipete had a different answer for that one, and we'll get to it in a second.

Camp four is admiring. Self-deprecating dev replies. Every time I see your workflow I realize how narrow my thinking is. They want a roadmap, even just a slice of this fleet, for their own projects.

And camp five, smallest but loudest, is just confused. People screenshotting parts of the post and asking, wait, what am I even looking at? The defaults Steipete uses are far enough out that some viewers literally can't parse the stack.

If you've ever scrolled past a tweet like this and felt one of those reactions in your gut, you already know which camp is yours. Notice that none of them are the camp the post is actually arguing for. Which is exactly why the next part matters.

And then Steipete dropped the line that re-frames everything.

When the cost-optimization camp pushed him on cheaper models and deterministic hooks, his reply was one sentence.

I could just disable fast mode and cut it down by seventy percent.

Fast mode is OpenAI Codex's one point five X speed tier. It burns credits faster in exchange for lower latency. So the spend isn't a constraint he's stuck with. It's a deliberate choice. It's not cost. It's latency. He's paying seventy percent more so the fleet stays fast enough to feel real-time.
